一、核心功能层面的问题
你的插件已经证明了基础识别和审计能力是有的。也就是说，它能够识别 192.168.85.1 -> 192.168.20.20:502 的 Modbus 流量，日志链路也建立起来了，这部分不是主要问题。
真正的问题在于阻断能力不稳定。
在 Default Action = Block 的场景下，虽然部分轮次里 FC03 和 FC16 会被拦截，但并不是每次都一致生效。更明显的是，在启用了“只允许读”或“只允许写”这类读写分类策略后，测试结果出现了明显不一致：同样配置下，有时读写都放行，有时又只有一部分生效。这说明插件当前还不能稳定实现“按功能码分类进行一致阻断”​。
换句话说，插件现在更像一个检测/审计插件，而不是一个已经完成验证的稳定阻断插件。
二、策略执行一致性的问题
报告里最突出的现象是：同一套配置，多轮测试结果不一致。
这类问题非常关键，因为白名单能力的价值就在于“确定性”。如果同样的配置在一次测试里能拦截、下一次又放过去了，那对于工业控制场景来说风险很大。工业白名单不能接受“偶尔有效”，它必须是“每次都按规则执行”。
目前你的插件存在以下一致性问题：
同一个策略下，FC03 和 FC16 的处置结果会波动。
读写分类策略没有形成稳定、可重复的业务侧效果。
偶发超时、通过、阻断切换并存，说明处置链路不是强确定性的。
这意味着插件当前不满足“同策略 = 同处置”的基本要求。
三、日志与真实业务结果不一致的问题
测试报告还反映出一个很大的问题：日志看到的结果和业务实际结果不同步。
例如有些轮次中，页面日志当时看不到明显记录，但切换页面后又能看到历史 pass 记录；同时 Ubuntu 服务端抓包又表明请求实际到达了应用层。这说明目前日志平面和真实数据平面的状态并不是完全一致的。
这会带来两个后果：
第一，运维人员难以判断策略是否真的生效。
第二，测试人员容易误判功能已经实现，但实际上只是日志记录了动作，真实流量并没有被稳定拦住。
所以从产品角度看，你的插件在“可观测性”和“结果可信度”上还有明显短板。
四、架构层面的问题
报告里已经明确给出了一个非常关键的背景：当前实现是旁路镜像检测架构，而不是严格的内联处置路径。
这其实就是很多现象的根因。
旁路镜像检测适合做识别、审计、告警，因为它本质上偏向“看见了什么”；但如果产品目标是工业协议白名单阻断，那么只靠旁路架构通常很难保证：
日志动作和真实转发动作完全一致；
每一条数据流都被严格拦截或放行；
策略变更后马上、稳定、无歧义地生效。
因此，你这个插件目前最大的架构问题是：检测能力和阻断目标之间存在设计落差。现有实现更适合“审计增强”，还不够支撑“严格白名单阻断”。
五、策略生效链路可能存在时序或状态残留问题
报告中还提到了几个很典型的信号，包括：
策略重载时序问题；
状态残留问题；
日志展示口径差异；
页面显示与实时处置不同步。
这说明插件在配置下发、规则生效、状态刷新、日志展示这整条链路上，可能还存在工程实现问题。也就是说，不一定只是“规则写得不对”，而是：
配置保存后没有立即完整作用到实际引擎；
旧状态没有被及时清理；
新旧规则切换过程中存在短暂窗口期；
页面展示拿到的是延迟数据，而不是实时处置结果。
这类问题会直接导致你看到的“这次拦住了、下次又没拦住”的现象。
六、协议分级控制能力还没有被真正证明
从测试结果来看，目前你能证明的是：
能识别 Modbus；
能记录日志；
能在某些情况下出现阻断现象。
但你还不能证明下面这些更高阶能力已经成立：
能稳定地区分读操作和写操作；
能稳定地只放行 FC03、阻断 FC16，或者反过来；
能在多轮重复测试中保持一致结果；
能把日志动作和真实业务动作严格对应起来。
所以如果从产品能力定义上说，当前版本还不能宣称已经完成了“工业协议白名单控制”，更准确的说法应该是：
已部分验证工业协议检测/审计能力，但尚未完成稳定阻断能力验证。​
七、界面与交互层面的问题
除了功能本身，报告还指出了两个比较明显的界面问题。
第一个是前置条件提示不会随状态变化自动消失。
在 IDS/IPS 已经启用且切换到 IPS 模式后，插件页面顶部仍然继续显示“需要启用 IDS/IPS”的提示。这会让用户怀疑到底有没有启用成功，也降低界面状态的可信度。
第二个是缺少“未 Apply 变更”的统一提示机制。
用户修改了配置但还没有点击 Apply 时，页面没有像 OPNsense 其他模块那样明确提示“有待应用的变更”。这很容易让人误以为设置已经生效，从而造成误操作或者错误测试结论。
这两个问题虽然不直接决定阻断能力，但会显著影响产品的专业感、可用性和测试准确性。
